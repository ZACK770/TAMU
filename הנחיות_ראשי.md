# מסמך אפיון דרישות טכנולוגי (PRD) – מערכת למידה ומבחנים אינטראקטיבית

מסמך זה מגדיר את הדרישות הטכנולוגיות, הארכיטקטורה וחוויית המשתמש עבור מערכת למידה ומבחנים רספונסיבית מבוססת TypeScript. המערכת מיועדת לעבודה על שרת יחיד ופורט יחיד, עם חיבור למסד נתונים PostgreSQL מרוחק.

## 1. ארכיטקטורת מערכת (TypeScript - שרת ופורט יחיד)

כדי לפשט את הפיתוח והתחזוקה, המערכת תיבנה כפרויקט TypeScript אחוד המאזין לפורט יחיד (לדוגמה 8080):

- **שפת פיתוח אחידה**: TypeScript (הן בבקאנד והן בפרונטאנד).
- **שרת אחרי (Backend)**: Node.js עם Express או Fastify ב-TypeScript. השרת אחראי על:
  - הגשת ה-API (נתיבי /api/*).
  - הגשת קבצי האתר הסטטיים (הפרונטאנד) מתוך תיקיית ה-Build (למשל dist או public).
- **צד לקוח (Frontend)**: React (באמצעות Vite) עם TypeScript ו-Tailwind CSS. האתר רספונסיבי לחלוטין (Responsive Web App) בגישת Mobile-First.
- **מסד נתונים (Database)**: PostgreSQL בענן בלבד (Supabase / Neon / Render). החיבור מתבצע באמצעות ORM תומך TypeScript (כמו Prisma או TypeORM).

## 2. חוויית המשתמש ומודולי האתר (User Experience Flow)

המערכת מחולקת למספר אזורים מרכזיים בעלי זרימה הדוקה ומאובטחת:

```
[ משתמש נכנס לאתר ] ──> [ דף הבית / אזור למידה וקבצים ]
                                │
                        (לחיצה על כניסה למבחן)
                                │
                   [ האם המשתמש מחובר ומאומת SMS? ]
                     ├──> לא ──> [ דף אימות טלפון (OTP) ] ──> [ דף הרשמה (במידת הצורך) ]
                     └──> כן ──> [ בדיקת מזהה ח"פ למבחן ]
                                           │
                              [ האם קיים מזהה מבחן תקין? ]
                                ├──> לא ──> [ מסך הזנת מזהה ידני ]
                                └──> כן ──> [ מעבר לממשק המבחן (צ'אט) ]
```

### א. דף הבית ופורטל הלימודים (Home Page & Portal)

- **עיצוב ומבנה**: דף בית מודרני, מרשים ונקי. כולל כותרת עליונה (Header) או תפריט צד (Sidebar) רספונסיבי המאפשר ניווט מהיר בין חלקי האתר.
- **תיאור הפעילות**: קטע המציג בצורה ברורה את מטרת הפעילות ותוכניות הלימוד השונות.
- **אזור למידה משותפת (Shared Learning)**:
  - נגן מדיה מובנה לצפייה בשיעורי וידאו או להאזנה לשיעורי שמע (Audio/Video Player).
  - אפשרות להורדת קבצי לימוד אקטואליים (קבצי PDF, חוברות עבודה) מתוך רשימה דינמית של תוכניות לימוד שונות.

### ב. מערך רישום, התחברות ואימות משתמש (Authentication via SMS/Voice)

- **אימות דו-שלבי (OTP)**: כל כניסה למערכת דורשת אימות מספר הטלפון של המשתמש.
- **אינטגרציית "ימות המשיח" (Yemot HaMashiach API)**:
  - בעת הזנת מספר טלפון, השרת מייצר קוד אימות חד-פעמי (למשל בן 4 או 6 ספרות).
  - השרת שולח קריאת API לימות המשיח לשליחת קוד ה-SMS או להפעלת שיחה קולית אוטומטית המקריאה את הקוד למשתמש (מתאים גם למכשירים כשרים).
  - המשתמש מזין את הקוד באתר, והשרת מייצר Token חתום (JWT) שנשמר בצד הלקוח (למשל ב-Cookie מאובטח) ומאמת אותו מול נתיבי ה-API.
- **רישום משתמשים חדשים**: במידה והטלפון אינו קיים במאגר, המערכת תעביר את המשתמש באופן אוטומטי למסך השלמת פרטים (שם מלא, עיר וכו') ורק לאחר מכן תאפשר לו להמשיך.
- **זמינות עתידית**: מבנה הנתונים יותאם להוספת התחברות מהירה באמצעות חשבון גוגל (Google Sign-In) בהמשך.

### ג. מנגנון כניסה למבחן ומזהה חד-פעמי (One-Time Token Validation)

- **חובת מזהה ייחודי**: כל מבחן מבוצע רק באמצעות מזהה ייחודי המורכב מספרות (Token). מזהה זה מבטיח כי כל נבחן לא יוכל לבצע את אותו המבחן מספר פעמים, או להשתמש בזהות אחרת עבור אותו המבחן.
- **זרימת כניסה דינמית**:
  - **כניסה דרך קישור (QR/URL)**: אם המשתמש נכנס דרך קישור המכיל את הפרמטר (למשל ?token=123456), המערכת קוראת את המזהה אוטומטית.
  - **כניסה ישירה**: אם המשתמש ניגש ישירות למסך המבחן ללא מזהה, המערכת תציג לו מקלדת נומרית מעוצבת ותדרוש ממנו להקיש את מספר המזהה שלו בצורה ידנית.
- **שיוך חד-חד-ערכי**: השרת מוודא במסד הנתונים שהמזהה קיים, תקין, ושייך למבחן המבוקש, וכן שהוא טרם נוצל על ידי המשתמש הנוכחי או משתמש אחר.

### ד. ממשק המבחן (Chat-Style Exam UI)

ממשק זה מעוצב באופן שונה לחלוטין משאר חלקי האתר ומדמה שיחת צ'אט אינטראקטיבית:

- **מיקוד מקסימלי**: חלון נקי נטול הסחות דעת (ללא תפריטים כלליים או קישורים חיצוניים).
- **אינטראקציה זורמת**:
  - השרת "שולח" את השאלה כבועת הודעה בצ'אט.
  - התשובות האמריקאיות מוצגות מתחת לשאלה ככפתורים מעוצבים ללחיצה קלה במובייל.
  - עם הלחיצה, התשובה שנבחרה נשלחת כהודעה של המשתמש בצ'אט, והשאלה הבאה מופיעה לאחר השהיה קלה (המעניקה תחושה של צ'אט חי וזורם).
  - ניתן להגדיר דרך ממשק הניהול האם השאלות יוצגו בסדר אקראי (מעורבב) עבור כל נבחן.

## 3. ממשקי API ואינטגרציות (TypeScript Routes)

להלן הגדרת הטיפוסים ונקודות הקצה המרכזיות בבקאנד, המאפשרות גם אינטגרציות חיצוניות פשוטות (כמו וואצאפ):

### א. נקודות קצה של אימות (Auth APIs)

- `POST /api/auth/send-code` - שליחת קוד אימות באמצעות ימות המשיח (SMS או שיחה קולית).
- `POST /api/auth/verify-code` - אימות הקוד שיוצר JWT סשן למשתמש.
- `POST /api/auth/register` - השלמת פרטי משתמש חדש.

### ב. נקודות קצה של המבחן (Exam APIs)

- `POST /api/exams/verify-token` - בדיקת תקינות מזהה המבחן הייחודי ושיוכו לסשן המשתמש.
- `GET /api/exams/session/:sessionId/next-question` - קבלת השאלה הבאה בצ'אט.
- `POST /api/exams/session/:sessionId/submit-answer` - שליחת תשובה לשאלה נוכחית.
- `POST /api/exams/session/:sessionId/complete` - נעילת המבחן וחישוב הציון הסופי.

## 4. ארכיטקטורת מסד נתונים (Prisma Schema Reference)

הנחיה עבור ה-AI ליצירת הטבלאות ב-PostgreSQL באמצעות Prisma:

```prisma
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

generator client {
  provider = "prisma-client-js"
}

model User {
  id          String        @id @default(uuid())
  phone       String        @unique
  fullName    String?
  createdAt   DateTime      @default(now())
  examSessions ExamSession[]
}

model Exam {
  id          String        @id @default(uuid())
  title       String
  isRandom    Boolean       @default(false)
  questions   Question[]
  tokens      ExamToken[]
}

model Question {
  id          String        @id @default(uuid())
  examId      String
  exam        Exam          @relation(fields: [examId], references: [id])
  text        String
  answers     String[]      // מערך תשובות
  correctIdx  Int           // אינדקס התשובה הנכונה
}

model ExamToken {
  id          String        @id @default(uuid())
  tokenCode   String        @unique // מזהה המבחן הייחודי (מספרי)
  examId      String
  exam        Exam          @relation(fields: [examId], references: [id])
  isUsed      Boolean       @default(false)
  usedById    String?
  usedAt      DateTime?
}

model ExamSession {
  id          String        @id @default(uuid())
  userId      String
  user        User          @relation(fields: [userId], references: [id])
  examId      String
  score       Int?
  answers     Json          // מעקב אחר תשובות המשתמש בזמן אמת
  isCompleted Boolean       @default(false)
  startedAt   DateTime      @default(now())
  completedAt DateTime?
}
```

## 6. מדריך פיתוח והרצה לעוזר ה-AI (AI Instruction Manual)

השתמש בהנחיות הבאות בכל פנייה ל-AI הפיתוח:

- **איסור מוחלט על סביבת DB מקומית**: תמיד יש לקרוא את משתנה הסביבה DATABASE_URL מהקובץ .env המחובר ישירות ל-PostgreSQL בענן. אין לכתוב פונקציות Fallback ל-SQLite או קבצים מקומיים.
- **מבנה קוד אחוד (Monorepo/Single Server)**:
  - קוד השרת (Node.js/Express) וקוד הלקוח (React) ישבו באותו מאגר.
  - קוד הפרונטאנד יעבור הידור לתיקיית dist, והשרת יגיש אותו באמצעות:
    ```typescript
    app.use(express.static(path.join(__dirname, '../frontend/dist')));
    app.get('*', (req, res) => {
      res.sendFile(path.join(__dirname, '../frontend/dist/index.html'));
    });
    ```
- **בדיקות מהירות מהטרמינל (Terminal Testing)**:
  - ודא כי כל ה-Endpoints מחזירים תשובות בפורמט JSON תקין ומאפשרים סימולציה מהטרמינל באמצעות פקודות curl לבדיקה מרחוק.

## 7. מערכת לוגינג (Logging System)

המערכת חייבת לכלול מערכת לוגינג מקיפה למעקב אחר פעילות המערכת:

- **ספריית לוגינג**: שימוש ב-winston או pino ללוגינג מקצועי
- **רמות לוגינג**: error, warn, info, debug
- **לוגינג לכל endpoint**: כל בקשת API תרשם עם:
  - זמן הבקשה
  - מתודה (GET/POST/PUT/DELETE)
  - נתיב
  - סטטוס תגובה
  - זמן עיבוד
  - מזהה משתמש (אם מחובר)
- **לוגינג לאירועים קריטיים**:
  - כניסות למערכת (מוצלחות וכושלות)
  - ניסיונות אימות טוקנים
  - שגיאות מערכת
  - קריאות ל-API חיצוני (ימות המשיח)
- **שמירת לוגים**: לוגים יישמרו בקבצים מסודרים בתיקיית `logs/` עם רוטציה יומית
- **לוגים רגישים**: לא לרשום סיסמאות, קודי OTP, או מידע רגיש אחר
